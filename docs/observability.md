# Observability

## Overview

llm-pipeline integrates with [OpenTelemetry](https://opentelemetry.io/) (OTel) via pydantic-ai's per-agent instrumentation. When enabled, each pipeline agent emits OTel spans for model requests, providing distributed tracing and token usage metrics without affecting any agents outside the pipeline.

Key capabilities:
- **Spans per model request**: each `agent.run_sync()` call produces an OTel span with model name, latency, and token counts
- **Token tracking**: input, output, and total token usage recorded per LLM call and aggregated per step
- **SQL-queryable cost data**: `PipelineStepState` persists token fields for offline cost analysis
- **Security-first defaults**: prompt/completion content is **not** sent to your OTel backend unless explicitly opted in

## Installation

Install the `[otel]` optional dependency group:

```bash
pip install llm-pipeline[otel]
```

This adds:
- `opentelemetry-sdk>=1.20.0`
- `opentelemetry-exporter-otlp-proto-http>=1.20.0`

The base `llm-pipeline` install does not pull in OTel packages. If you pass `instrumentation_settings` without the `[otel]` group installed, pydantic-ai raises an `ImportError` at agent construction time.

## Configuration

Pass an `InstrumentationSettings` instance to `PipelineConfig` via the `instrumentation_settings` parameter. Every agent built by the pipeline will use this configuration.

```python
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
from pydantic_ai import InstrumentationSettings

# 1. Set up a TracerProvider with OTLP exporter
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter())
)

# 2. Create InstrumentationSettings
instrumentation = InstrumentationSettings(tracer_provider=provider)

# 3. Pass to pipeline
pipeline = MyPipeline(
    model="google-gla:gemini-2.0-flash-lite",
    instrumentation_settings=instrumentation,
)
```

When `instrumentation_settings` is `None` (the default), no OTel instrumentation is applied and no OTel imports are required at runtime.

## Content Capture (include_content)

By default, `InstrumentationSettings` does **not** include prompt or completion content in spans. This prevents sensitive data from leaking to telemetry backends.

To opt in:

```python
instrumentation = InstrumentationSettings(
    tracer_provider=provider,
    include_content=True,  # sends prompts + completions to OTel backend
)
```

Only enable this in development or when your OTel backend has appropriate access controls. Production deployments should leave `include_content=False` (the default).

## Token Tracking

### Per-step database fields

After each step executes, `PipelineStepState` records token usage in four nullable columns:

| Column | Type | Description |
|--------|------|-------------|
| `input_tokens` | `int \| None` | Input tokens consumed by this step |
| `output_tokens` | `int \| None` | Output tokens generated by this step |
| `total_tokens` | `int \| None` | `input_tokens + output_tokens`, pre-computed for efficient SQL aggregation |
| `total_requests` | `int \| None` | Number of LLM requests made (includes consensus attempts) |

Values are `None` when the model provider does not report usage data.

### SQL cost aggregation

Because `total_tokens` is stored as a pre-computed column, cost queries require no application logic:

```sql
-- Total tokens per pipeline run
SELECT run_id, SUM(total_tokens) AS run_tokens, SUM(total_requests) AS run_requests
FROM pipeline_step_states
WHERE total_tokens IS NOT NULL
GROUP BY run_id;

-- Per-step breakdown for a specific run
SELECT step_name, input_tokens, output_tokens, total_tokens, total_requests
FROM pipeline_step_states
WHERE run_id = 'your-run-id'
ORDER BY step_number;

-- Daily token spend across all pipelines
SELECT DATE(created_at) AS day, SUM(total_tokens) AS daily_tokens
FROM pipeline_step_states
WHERE total_tokens IS NOT NULL
GROUP BY DATE(created_at);
```

### Database migration

The four token columns are added automatically via `ADD COLUMN` migration when `init_pipeline_db()` runs against an existing SQLite database. No manual migration step is needed.

> **Non-SQLite databases:** The auto-migration uses SQLite-specific `PRAGMA table_info` and is skipped for other engines. If you use PostgreSQL or another backend, add the four columns (`input_tokens`, `output_tokens`, `total_tokens`, `total_requests` -- all nullable `INTEGER`) manually.

## Event Fields

Two event types carry token data, providing different granularity levels.

### LLMCallCompleted (per-call granularity)

Emitted after each individual `agent.run_sync()` call. Fields:

| Field | Type | Description |
|-------|------|-------------|
| `input_tokens` | `int \| None` | Tokens consumed by this single call |
| `output_tokens` | `int \| None` | Tokens generated by this single call |
| `total_tokens` | `int \| None` | Sum of input + output for this call |

In a consensus step that makes 5 attempts, you receive 5 `LLMCallCompleted` events, each with its own token values.

### StepCompleted (step-aggregate granularity)

Emitted once per step after all calls (including consensus retries) finish. Fields:

| Field | Type | Description |
|-------|------|-------------|
| `input_tokens` | `int \| None` | Sum of input tokens across all calls in this step |
| `output_tokens` | `int \| None` | Sum of output tokens across all calls in this step |
| `total_tokens` | `int \| None` | Sum of total tokens across all calls in this step |

The `StepCompleted` totals match what is persisted to `PipelineStepState`, giving you the true cost of the step including all consensus attempts.

> **Cached steps:** When a step is served from cache, `StepCompleted` is emitted with all token fields set to `None` because no LLM calls were made. If you filter or aggregate events by token fields, account for `None` values on cached steps.

## Example: Console Span Exporter

Minimal working example that prints spans to stdout:

```python
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import SimpleSpanProcessor, ConsoleSpanExporter
from pydantic_ai import InstrumentationSettings

# Set up console exporter (prints spans to stdout)
provider = TracerProvider()
provider.add_span_processor(
    SimpleSpanProcessor(ConsoleSpanExporter())
)

instrumentation = InstrumentationSettings(tracer_provider=provider)

# Use with your pipeline
pipeline = MyPipeline(
    model="google-gla:gemini-2.0-flash-lite",
    instrumentation_settings=instrumentation,
)

pipeline.context["input_text"] = "Hello, world!"
pipeline.execute()

# Spans print to stdout as each agent.run_sync() completes
```

## Environment Variables

Standard OTel environment variables are respected by the OTLP exporter. No llm-pipeline-specific env vars are needed.

| Variable | Purpose | Example |
|----------|---------|---------|
| `OTEL_EXPORTER_OTLP_ENDPOINT` | OTLP collector endpoint | `http://localhost:4318` |
| `OTEL_EXPORTER_OTLP_HEADERS` | Auth headers for collector | `Authorization=Bearer token` |
| `OTEL_SERVICE_NAME` | Service name in traces | `my-pipeline-service` |
| `OTEL_RESOURCE_ATTRIBUTES` | Additional resource attributes | `deployment.environment=prod` |

When using the OTLP exporter, these variables configure the exporter automatically if not set programmatically:

```python
import os
os.environ["OTEL_EXPORTER_OTLP_ENDPOINT"] = "http://localhost:4318"
os.environ["OTEL_SERVICE_NAME"] = "my-pipeline-service"

# OTLPSpanExporter() picks up the env vars
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
exporter = OTLPSpanExporter()  # uses OTEL_EXPORTER_OTLP_ENDPOINT
```
