# PLANNING

## Summary

Enable pydantic-ai OTel instrumentation per pipeline agent via `instrument=` parameter in `build_step_agent()`, capture token usage from `RunResult.usage()` after every `agent.run_sync()` call, store aggregate tokens on `PipelineStepState` (new nullable fields + ADD COLUMN migration), enrich `LLMCallCompleted` and `StepCompleted` events with token data, add `[otel]` optional dependency group to pyproject.toml, and write docs/observability.md. No new event types needed -- the event system (28 types) already exists.

## Plugin & Agents

**Plugin:** python-development
**Subagents:** backend-development
**Skills:** none

## Phases

1. **Schema changes**: Add token fields to `PipelineStepState` (state.py) and `LLMCallCompleted` / `StepCompleted` event types (events/types.py)
2. **Agent instrumentation**: Add `instrument=` parameter threading from `build_step_agent()` through `PipelineConfig`
3. **Token capture**: Capture `run_result.usage()` in both normal and consensus paths in pipeline.py; propagate to events and state
4. **Dependencies and docs**: Add `[otel]` group to pyproject.toml; create docs/observability.md

## Architecture Decisions

### Per-Agent vs Global Instrumentation
**Choice:** Per-agent `instrument=` parameter on `Agent(...)` constructor in `build_step_agent()`
**Rationale:** CEO decision Q1. Scopes OTel to pipeline agents only. Prevents side effects on consumer's own pydantic-ai agents. `InstrumentationSettings` threaded as optional param through `PipelineConfig.__init__` -> `build_step_agent()`.
**Alternatives:** `Agent.instrument_all()` -- rejected; process-wide, cannot be undone per-agent

### include_content Default
**Choice:** `include_content=False` default on any internally-constructed `InstrumentationSettings`
**Rationale:** CEO decision Q2. Security-first: prompts and completions not sent to OTel backend unless consumer explicitly opts in by passing a custom `InstrumentationSettings` with `include_content=True`.
**Alternatives:** Default True (pydantic-ai default) -- rejected; sends prompt content to telemetry backends by default

### Token Aggregation in Consensus
**Choice:** Sum tokens across all consensus attempts for `PipelineStepState`; per-call tokens on each `LLMCallCompleted`
**Rationale:** CEO decision Q3. `PipelineStepState` reflects true cost of the step. Per-call granularity preserved on individual `LLMCallCompleted` events.
**Alternatives:** Winner-only tokens -- rejected; underreports actual cost

### Field Naming Convention
**Choice:** `input_tokens`, `output_tokens`, `total_tokens` (+ `total_requests` for `PipelineStepState`)
**Rationale:** CEO decision Q4. Matches pydantic-ai `RunUsage` API (`input_tokens`, `output_tokens`, `requests`). Consistent with OTel gen_ai semantic conventions.
**Alternatives:** `request_tokens` / `response_tokens` (task description names) -- rejected; don't match pydantic-ai API

### OTel Dependency Group
**Choice:** New `[otel]` optional group in pyproject.toml; also added to `[dev]`
**Rationale:** CEO decision Q5. Keeps base install light. OTel is fully opt-in. Requires `opentelemetry-sdk>=1.20.0`; exporter (`opentelemetry-exporter-otlp-proto-http>=1.20.0`) included as convenience.
**Alternatives:** Bundle with `[pydantic-ai]` group -- rejected; forces OTel on all pydantic-ai users

### total_tokens Storage
**Choice:** Stored as computed field `total_tokens = input_tokens + output_tokens` at write time on `PipelineStepState`
**Rationale:** CEO decision Q6. Enables efficient SQL aggregation (`SUM(total_tokens)`) for cost queries without application-layer arithmetic.
**Alternatives:** Derive on read -- rejected; requires JOIN/subquery for aggregate cost queries

### DB Migration Strategy
**Choice:** ADD COLUMN with nullable int fields on `PipelineStepState`; no migration framework, use SQLite-compatible `ADD COLUMN IF NOT EXISTS` pattern
**Rationale:** Consistent with existing `SQLiteEventHandler.__init__` migration pattern in handlers.py. SQLite supports ADD COLUMN but not IF NOT EXISTS -- use `PRAGMA table_info` check or try/except pattern to match existing codebase.
**Alternatives:** Alembic -- out of scope for this library; schema version field -- not present in codebase pattern

## Risks & Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| `frozen=True, slots=True` dataclass field ordering breaks on adding optional fields to `LLMCallCompleted` / `StepCompleted` | High | All new fields use `field(default=None)` -- ensure placed AFTER required fields; `slots=True` does not affect field ordering, only memory layout |
| `InstrumentationSettings` import path changed across pydantic-ai versions | Medium | Verify import path via Context7 `/pydantic/pydantic-ai/v1_0_5` before coding; guard with `TYPE_CHECKING` to avoid runtime import error |
| `run_result.usage()` returns None for some model providers | Low | Defensive guard: `_usage = run_result.usage() or None`; all token fields remain `Optional[int]` |
| ADD COLUMN migration silently fails on already-migrated DB | Medium | Use `PRAGMA table_info` to check column existence before `ALTER TABLE`; follow handlers.py pattern exactly |
| Consensus path refactor (return tuple) breaks existing callers | Medium | `_execute_with_consensus` is a private method; only one call site in `execute()`; update both simultaneously in Step 6 |
| `total_tokens` computed twice (caller and `_save_step_state`) | Low | Compute once in caller (Steps 5/6); `_save_step_state` receives pre-computed value; add guard to avoid double-compute |
| OTel deps not installed when `InstrumentationSettings` is passed | Medium | Document clearly in observability.md; pydantic-ai will raise an informative ImportError at Agent construction time |

## Phase Recommendation

**Risk Level:** medium
**Reasoning:** Core logic changes are contained to private methods in pipeline.py, but the DB migration (ADD COLUMN) and frozen dataclass field additions carry moderate risk. The consensus return-type refactor is a private method but touches runtime behavior. Schema changes are additive (nullable columns) so backward compat is preserved. No new event types needed. OTel activation is opt-in and guarded behind a None check.
