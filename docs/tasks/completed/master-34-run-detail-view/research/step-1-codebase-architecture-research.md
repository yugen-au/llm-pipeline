## Route Patterns

### File-based routing (TanStack Router)
- Routes defined via `createFileRoute(path)` in `src/routes/`
- Auto-generated `routeTree.gen.ts` by `@tanstack/router-plugin/vite`
- Vite plugin: `tanstackRouter({ autoCodeSplitting: true })`
- Search params validated with Zod via `@tanstack/zod-adapter`

### Existing route: `runs/$runId.tsx` (placeholder)
```typescript
const runDetailSearchSchema = z.object({
  tab: fallback(z.string(), 'steps').default('steps'),
})

export const Route = createFileRoute('/runs/$runId')({
  validateSearch: zodValidator(runDetailSearchSchema),
  component: RunDetailPage,
})

function RunDetailPage() {
  const { runId } = Route.useParams()
  const { tab } = Route.useSearch()
  // ... placeholder content
}
```

### Pattern from `index.tsx` (reference)
- Named function exports (not arrow functions)
- Search schema outside component, Route export uses `createFileRoute`
- Component reads params/search via `Route.useParams()` / `Route.useSearch()`
- Data fetching via TanStack Query hooks (not loaders)
- Navigation via `useNavigate()` from `@tanstack/react-router`

## Component Patterns

### Naming & Export
- Named function exports: `export function ComponentName() {}`
- Private sub-components: plain `function SubComponent()` (no export)
- Props interface defined above component: `interface ComponentProps { ... }`

### Styling
- Tailwind utility classes exclusively (no CSS modules, no CSS-in-JS)
- shadcn/ui components for primitives (Badge, Button, Table, etc.)
- `cn()` from `@/lib/utils` for conditional class merging
- Design tokens via CSS variables: `bg-background`, `text-foreground`, `text-muted-foreground`, `border-sidebar-border`, etc.
- Dark mode: class-based (`dark` on `<html>`)

### Layout
- Root: `flex h-screen bg-background` with sidebar (w-60) + `flex-1 overflow-auto` main
- Pages: `p-6` padding, `gap-4` spacing between sections
- Semantic classes: `bg-card`, `text-card-foreground`, `rounded-lg border`

### Loading/Error/Empty States
- Skeleton rows with `animate-pulse rounded bg-muted` divs
- Error: `text-destructive` centered text
- Empty: `text-muted-foreground` centered text
- All 3 states handled in the same component via conditional rendering

### Path Alias
- `@/` maps to `src/` via tsconfig paths + Vite resolve.alias

## Data Fetching Patterns

### TanStack Query Hooks
- Located in `src/api/*.ts`
- Use `apiClient<T>()` from `./client`
- Use `queryKeys` factory from `./query-keys`
- Dynamic `staleTime`: `Infinity` for terminal runs, shorter for active

### Hooks Available for Task 34

| Hook | Params | Returns | Notes |
|------|--------|---------|-------|
| `useRun(runId)` | `string` | `RunDetail` | includes `steps: StepSummary[]`, dynamic staleTime |
| `useRunContext(runId, status?)` | `string, RunStatus?` | `ContextEvolutionResponse` | `snapshots: ContextSnapshot[]` |
| `useSteps(runId, runStatus?)` | `string, RunStatus?` | `StepListResponse` | `items: StepListItem[]`, polls 3s for active |
| `useStep(runId, stepNumber, runStatus?)` | `string, number, RunStatus?` | `StepDetail` | full step detail |
| `useEvents(runId, filters, runStatus?)` | `string, EventListParams, RunStatus?` | `EventListResponse` | `items: EventItem[]` |
| `useWebSocket(runId \| null)` | `string \| null` | `{ status, error }` | WS + cache integration |

### Query Key Hierarchy
```
['runs']                          -- all runs
['runs', filters]                 -- filtered list
['runs', runId]                   -- single run detail
['runs', runId, 'context']        -- context evolution
['runs', runId, 'steps']          -- step list
['runs', runId, 'steps', stepNum] -- step detail
['runs', runId, 'events', filters]-- event list
```

## Backend Event Types (for step detail tabs context)

Categories with event types:
- **pipeline_lifecycle**: PipelineStarted, PipelineCompleted, PipelineError
- **step_lifecycle**: StepSelecting, StepSelected, StepSkipped, StepStarted, StepCompleted
- **cache**: CacheLookup, CacheHit, CacheMiss, CacheReconstruction
- **llm_call**: LLMCallPrepared, LLMCallStarting, LLMCallCompleted, LLMCallRetry, LLMCallFailed, LLMCallRateLimited
- **consensus**: ConsensusStarted, ConsensusAttempt, ConsensusReached, ConsensusFailed
- **instructions_context**: InstructionsStored, InstructionsLogged, ContextUpdated
- **transformation**: TransformationStarting, TransformationCompleted
- **extraction**: ExtractionStarting, ExtractionCompleted, ExtractionError
- **state**: StateSaved

Key event fields for task 35's tabs (informational for task 34):
- `LLMCallStarting.event_data`: `{ rendered_system_prompt, rendered_user_prompt, call_index }`
- `LLMCallCompleted.event_data`: `{ raw_response, parsed_result, model_name, attempt_count, validation_errors }`
- `ContextUpdated.event_data`: `{ new_keys, context_snapshot }`
- `ExtractionCompleted.event_data`: `{ extraction_class, model_class, instance_count, execution_time_ms }`

Events are step-scoped via `step_name` field in `event_data`. Events with `event_type.includes('step')` are step-scoped per websocket.ts logic.

## Architecture Decisions (Pre-validated)

1. **Step selection via Zustand**: useUIStore.selectStep(stepNumber) -- already designed for this
2. **No URL param for selected step**: ephemeral state, resets on page reload (existing pattern)
3. **`tab` search param**: already in the route schema, will be used by task 35's panel tabs
4. **Dynamic staleTime**: all hooks support passing runStatus for terminal optimization
5. **WebSocket integration**: useWebSocket(runId) can be wired for live run updates
