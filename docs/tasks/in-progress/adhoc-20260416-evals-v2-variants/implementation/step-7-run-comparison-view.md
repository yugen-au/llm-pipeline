# IMPLEMENTATION - STEP 7: RUN COMPARISON VIEW
**Status:** completed

## Summary
Built frontend comparison view at `/evals/{datasetId}/compare` (search params `baseRunId`, `variantRunId`) rendering two runs side-by-side: header w/ variant name lookup, delta_snapshot via JsonViewer, stat cards w/ color-coded deltas, and per-case pass/fail table w/ delta indicator. ## Changes

### File: `llm_pipeline/ui/frontend/src/routes/evals.$datasetId.compare.tsx`
New file-based route w/ `validateSearch` via `zodValidator` coercing `baseRunId`/`variantRunId` string search params to positive integers (fallback 0 triggers invalid-URL state). `CompareRunsPage`:
- Fetches both runs in parallel via `useEvalRun(datasetId, id)`.
- If variant run has `variant_id`, looks up variant metadata via `useVariant(datasetId, variantIdForLookup)` for name display.
- Loading, error, and invalid-params branches each render back-to-dataset link.
- Layout: breadcrumb -> header -> two run-summary cards (status, timestamp, deep-link to run detail) -> delta-summary card (JsonViewer `data=` mode on `delta_snapshot`; falls back to "no snapshot") -> 4 stat cards (passed, failed, errored, pass-rate) w/ side-by-side baseline/variant values plus color-coded `DeltaBadge`/`DeltaPctBadge` (green when improvement, red when regression; `invertColor` for failed/errored where lower is better) -> per-case comparison table.
- Per-case table: union of case names (sorted), columns = Case / Baseline pass-fail Badge / Baseline scores / Variant pass-fail Badge / Variant scores / Delta indicator. `PassFailBadge` handles N/A / errored / pass / fail. `caseDelta()` returns improved/regressed/unchanged/n/a from pre-computed `passed` field (errored treated as not-passed).
- `ScoresCell` extracts numeric/boolean scores from `evaluator_scores` (handles both raw and `{value}` shapes per existing run-detail pattern).

## Decisions

### Baseline selection on variant run detail page
**Choice:** Fetch the full run list via `useEvalRuns(datasetId)` and pick the most recent run with `variant_id == null` (excluding the current run) by `started_at` descending, client-side.
**Rationale:** Contract says "most recent baseline run" and accepts fetch-on-click for simplicity. Reusing `useEvalRuns` avoids a new backend endpoint; list is already keyed by dataset and cached by React Query. Completed-at/status is not filtered because "most recent" is the stated criterion; a future iteration could filter to completed runs only.
**Alternatives:** Add a dedicated `/runs?variant_id__isnull=true&limit=1` filter — rejected, heavier + requires backend change.

### Delta color semantics for failed/errored stats
**Choice:** `invertColor` prop on `ComparisonStatCard` / `DeltaBadge` so that a decrease in failed/errored renders green (improvement).
**Rationale:** Semantic correctness — more passes, fewer failures/errors is universally "improvement"; surfacing that directly avoids user confusion.
**Alternatives:** Always color red=positive, green=negative — rejected, misleading.

### Search-param validation
**Choice:** `zodValidator` w/ `z.coerce.number().int().positive()` and `fallback(..., 0)` default 0; component renders invalid-URL state when either is 0.
**Rationale:** Matches existing pattern in `routes/runs/$runId.tsx`; coerce handles `?baseRunId=5&variantRunId=7` URL strings; fallback avoids throwing on missing/malformed params, letting us show a friendly error w/ back link.
**Alternatives:** Custom `validateSearch` function — rejected, `zodValidator` already used elsewhere in the codebase.

### Variant name lookup
**Choice:** Fetch `useVariant(datasetId, variantIdForLookup)` only after variant run loads; pass 0 when variant_id is null so hook is disabled (`enabled: datasetId > 0 && variantId > 0` already exists).
**Rationale:** Avoids a conditional hook call; leverages existing hook gating. Falls back to "#<id>" when variant no longer exists (e.g. deleted since run).

### Step 6 integration boundary respected
**Choice:** Did NOT touch `evals.$datasetId.index.tsx` (Run History tab). Per-step-7 contract notes, bulk-select "Compare" in Run History is a Step 6 concern / follow-up.
**Rationale:** Contract-forbidden file. Deep-link URL + Compare button on variant run detail fulfil the user-facing Step-7 requirement without encroaching on Step 6.
