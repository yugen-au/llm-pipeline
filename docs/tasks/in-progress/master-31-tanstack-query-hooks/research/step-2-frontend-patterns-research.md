## 7. WebSocket Architecture

### Endpoint: `ws://host/ws/runs/{run_id}`

### Connection Flow
1. Client connects with run_id
2. Server checks run status:
   - **Not found**: sends `{ type: "error", detail: "Run not found" }`, closes with 4004
   - **Completed/failed**: replays all persisted events, sends `{ type: "replay_complete", run_status, event_count }`, closes with 1000
   - **Running**: connects to live queue, streams events in real-time

### Message Types (server -> client)
| type | Fields | When |
|------|--------|------|
| `heartbeat` | `timestamp` | Every 30s of inactivity |
| `stream_complete` | `run_id` | Live run finished |
| `replay_complete` | `run_status`, `event_count` | After replaying completed run events |
| `error` | `detail` | Run not found |
| (raw event) | varies per event type | Each pipeline event during execution |

### Close Codes
- 1000: Normal (replay complete)
- 4004: Run not found
- 1011: Server error

### ConnectionManager Pattern
- Per-client `thread_queue.Queue` fan-out (sync `put_nowait` from pipeline code)
- `manager.broadcast_to_run(run_id, event_data)` for live events
- `manager.signal_run_complete(run_id)` sends None sentinel
- `_stream_events()` async wrapper: `asyncio.to_thread(queue.get, True, timeout)` with heartbeat on Empty

## 11. Query Key Strategy Recommendation

Based on the API surface, recommended query key hierarchy:
```
['runs', { filters }]              -> run list
['runs', runId]                    -> run detail
['runs', runId, 'context']         -> context evolution
['runs', runId, 'steps']           -> step list
['runs', runId, 'steps', stepNum]  -> step detail
['runs', runId, 'events', { filters }] -> event list
['prompts', { filters }]           -> prompt list (future)
['pipelines']                      -> pipeline list (future)
['pipelines', name]                -> pipeline detail (future)
```

This enables hierarchical invalidation: `invalidateQueries({ queryKey: ['runs'] })` clears all run-related data, `invalidateQueries({ queryKey: ['runs', runId] })` clears a specific run + its children.

## 12. Caching Strategy Observations

- Default `staleTime: 30_000` (30s) is already configured globally
- Active/running runs need shorter staleTime or polling (`refetchInterval`)
- Completed runs can use longer staleTime (data won't change)
- WebSocket events should update query cache directly via `queryClient.setQueryData` for live runs
- POST /api/runs mutation should invalidate `['runs']` list query
